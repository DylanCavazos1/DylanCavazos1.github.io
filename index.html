Course Manager


The Course Manager project from my CS 300 Data Structures and Algorithms course was the project that I selected for each of the three artifacts and enhancments. This program was initially developed in C++ and would load data from a CSV file, containing the relevant student course information such as the course ID, course Title, and any prerequisites. I selected this project for my artifact because I wanted to enhance the overall software design of the program, improve the performance of loading and querying course data, introduce CRUD operations for user management, and create a seamless experience for a user.

Software Design and Engineering

Originally, this program loaded course data from a CSV file and utilized a Hash Table to store the courses for retrieval and management. Allowing a user to search by course ID and print an individual course and view the course ID, course Title, and prerequisites. I felt this was a great opportunity to convert the original C++ code to a more modern language like Python, focusing on improving the readability and maintability of the program. For this enhancement, I solely focused on building the program from the ground up in a new language, implement loading of course data into a dictionary, and seperate the core applications logic into multiple classes for a more modular design.

Data Structure and Algorithms

Originally, the Course Manager application was developed using a Hash-Table algorithm, in which a has function would be computed that would be used in storing the course ID in a linked-list. If the linked-list contained a node that was already occupied by a current course, then a while loop would iterate through the linked-list until an empty node was found and would then place the course ID there for retrieval and storage. I chose this for my Data Structures and Algorithm artifact to improve the search efficiency by utilizing a Binary Search Tree instead of a Hash-Table. This would improve the overall time complexity for searching and insertion since it doesn't suffer from collisions.

Databases

The Course Manager project initially loaded course data from a CSV file. This required the CSV data to be loaded each time the application was used and also required an absolute path when loading the courses to the Hash Table. This also meant that any changes to the course data was not persistent and would require reloading upon restarting the application. In my enhancement for the Database artifact I introduced a relational database to store the course data, and because I converted the program to Python I selected a PostgreSQL database to interface directly with the Python application using the psycopg2 library. This enabled more secure and persistent data storage, while spatially allowing a user to load course data from a CSV file should they choose.
